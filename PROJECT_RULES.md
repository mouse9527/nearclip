# NearClip 项目开发规则

## 项目简介

**NearClip** 是一个基于近场通信技术的多平台跨设备剪贴板同步解决方案。该项目旨在为用户提供无缝的跨设备内容共享体验，支持在不同设备间快速同步剪贴板内容。

### 核心特性
- **设备配对**: 多端设备间需要配对授权后才可建立通信连接，确保数据安全
- **近场通信**: 采用蓝牙低功耗 (BLE) 技术实现设备间的近距离通信
- **智能感知**: 已配对的两个设备靠近后自动检测并同步剪贴板内容
- **多平台支持**: 原生支持 Android 和 macOS 平台
- **实时同步**: 剪贴板内容变化时自动同步到配对设备
- **安全可靠**: 端到端加密保护数据传输安全
- **低功耗设计**: 优化的通信协议确保最小电量消耗

### 技术架构
- **核心引擎**: Rust 实现的高性能异步通信库
- **平台绑定**: C 兼容接口支持多平台集成
- **用户界面**: 各平台原生 UI 框架 (Android Compose + macOS SwiftUI)
- **通信协议**: 自定义的轻量级数据同步协议

### 工作流程
1. **初次配对**: 用户在两台设备上分别启动应用，通过扫描或输入配对码完成设备授权
2. **自动发现**: 已配对设备在蓝牙范围内时自动相互发现并建立连接
3. **内容监听**: 应用监听系统剪贴板变化事件
4. **智能同步**: 检测到剪贴板内容变化时，自动加密并传输到配对设备
5. **无缝体验**: 用户在任一设备复制内容后，可在另一设备直接粘贴使用

### 应用场景
- 开发者在多设备间快速共享代码片段
- 设计师在不同设备间传输创意内容
- 日常用户实现手机与电脑间的无缝内容流转
- 团队协作中的快速信息分享

## 极限编程 (XP) 实践

### 1. 结对编程 (Pair Programming)
- **强制要求**: 所有核心功能开发必须采用结对编程
- **适用范围**: Rust 核心库、平台绑定层、关键业务逻辑
- **例外情况**: 文档更新、配置调整、简单 bug 修复可单人完成
- **工具支持**: 使用 `scripts/pair_gui.py` 进行配对会话管理

### 2. 测试驱动开发 (TDD)
- **红-绿-重构循环**: 严格遵循 TDD 三步法
- **测试覆盖率要求**:
  - Rust 核心库: ≥ 90%
  - 平台绑定: ≥ 80%
  - UI 层: ≥ 70%
- **测试类型**:
  - 单元测试: 每个公共函数必须有对应测试
  - 集成测试: 跨模块交互测试
  - 端到端测试: 完整用户场景测试

### 3. 持续集成 (CI)
- **自动化构建**: 每次 push 触发完整构建流程
- **多平台验证**: 同时验证 Android 和 macOS 构建
- **质量门禁**: 所有检查通过才能合并到 master

### 4. 小步提交 (Small Releases)
- **提交频率**: 每个功能点完成后立即提交
- **提交信息格式**:
  ```
  <type>(<scope>): <description>
  
  [optional body]
  
  [optional footer]
  ```
- **类型定义**: feat, fix, docs, style, refactor, test, chore

### 5. 简单设计 (Simple Design)
- **YAGNI 原则**: 不实现当前不需要的功能
- **重构优先**: 优先重构现有代码而非添加新功能
- **接口最小化**: API 设计遵循最小接口原则

## 代码质量硬性限制

### Rust 代码标准

#### 1. 编译器检查
```toml
# 必须在 Cargo.toml 中启用
[lints.rust]
warnings = "deny"
unused = "deny"
dead_code = "deny"

[lints.clippy]
all = "deny"
pedantic = "deny"
```

#### 2. 代码复杂度限制
- **方法参数限制**: 方法参数不得超过 3 个，超过时使用对象封装
- **方法长度限制**: 单个方法不得超过 50 行代码（不含注释和空行）
- **类长度限制**: 单个类不得超过 200 行代码（不含注释和空行）
- **方法复杂度**: 圈复杂度不得超过 10
- **嵌套深度**: 代码嵌套层级不得超过 4 层
- **类职责**: 每个类只能有一个职责（单一职责原则）
- **方法职责**: 每个方法只做一件事情

#### 3. 代码规范
- **变量命名**: 变量名必须具有明确的业务含义，禁止使用 a、b、c 等无意义命名
- **魔法数字**: 禁止使用魔法数字，必须定义为常量
- **代码重复**: 重复代码不得超过 3 行，必须提取为方法或常量

#### 4. 内存安全
- **禁止 unsafe**: 除非有充分理由并经过 code review
- **生命周期标注**: 所有引用必须明确生命周期
- **错误处理**: 禁止使用 `unwrap()` 和 `expect()`，必须使用 `?` 或 `match`
- **异常处理**: 不允许空的 catch 块，必须记录日志或重新抛出

#### 5. 性能要求
- **异步优先**: 网络 I/O 必须使用 async/await
- **零拷贝**: 大数据传输优先使用引用
- **内存分配**: 避免不必要的 `clone()` 和 `to_string()`

### 平台代码标准

#### Android (Kotlin)
```kotlin
// ktlint 配置
ktlint {
    android = true
    outputToConsole = true
    ignoreFailures = false
}
```
- **空安全**: 严格使用可空类型
- **协程**: 异步操作必须使用协程
- **Compose**: UI 组件必须是无状态的

#### macOS (Swift)
```swift
// SwiftLint 规则
swiftlint:
  rules:
    - line_length: 120
    - function_body_length: 50
    - cyclomatic_complexity: 10
```
- **内存管理**: 避免循环引用，正确使用 weak/unowned
- **并发**: 使用 async/await 处理异步操作
- **错误处理**: 使用 Result 类型或 throws

### 构建和部署要求

#### 1. 构建脚本
- **Just 命令**: 所有构建操作必须通过 `justfile` 定义
- **依赖检查**: 构建前自动检查依赖版本
- **清理构建**: 每次发布前执行 `cargo clean`

#### 2. 版本管理
- **语义化版本**: 严格遵循 SemVer
- **变更日志**: 每个版本必须更新 CHANGELOG.md
- **标签管理**: 发布版本必须打 git tag

#### 3. 安全要求
- **依赖审计**: 定期运行 `cargo audit`
- **密钥管理**: 禁止硬编码密钥和敏感信息
- **权限最小化**: 应用权限申请遵循最小权限原则

## 违规处理

### 警告级别
1. **轻微违规**: 代码格式、注释不规范
2. **中等违规**: 测试覆盖率不足、复杂度超标
3. **严重违规**: 内存安全问题、性能严重回归

### 处理流程
1. 自动检查失败 → 阻止合并
2. 人工 Review 发现问题 → 要求修改
3. 生产环境问题 → 立即回滚 + 事后分析
